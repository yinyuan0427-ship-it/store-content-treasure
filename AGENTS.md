# AGENTS.md — 门店内容宝项目协作说明

本文档给后续接手本项目的 Codex、Claude Code 或其他工程代理使用。请先读完再修改代码，避免误改业务方向。

## 1. 项目定位

这是一个面向 TEMPUR / 床垫 / 软体家居经销商门店的手机 H5 项目，当前名称倾向于“门店内容宝 / 门店案例宝”。

核心目标不是做普通素材站，而是帮助门店把真实交付案例、产品资料、导购内容和客户留资串成一个闭环：

1. 导购或门店创建交付案例采集任务。
2. 安装师傅上传真实安装 / 交付照片。
3. 导购补充成交故事、客户反馈和适合公开的内容。
4. 管理员审核案例，确认隐私风险、重复图和公开权限。
5. 审核通过后沉淀为门店案例资产。
6. 导购可把公开案例分享给客户，客户可查看案例并留资。
7. 产品资料库用于导购快速查产品卖点、适合人群、痛点和详情。
8. 素材库提供朋友圈、小红书等可复制内容，但不应该混入客户案例和送货实拍。

当前用户已经开始“真实试用”，所以后续开发要优先考虑数据同步、稳定性和隐私合规，而不仅是本地演示效果。

## 2. 技术栈与运行方式

- 框架：React 18 + TypeScript
- 构建：Vite 5
- 路由：react-router-dom v6
- 样式：Tailwind CSS
- 图标：lucide-react
- 当前数据：大量 mock 数据 + localStorage 持久化
- 部署：Vercel 静态 SPA，`vercel.json` 已配置路由回退

常用命令：

```bash
npm.cmd run dev
npm.cmd exec tsc -- --noEmit
npm.cmd run build
```

注意：`npm run build` 实际执行 `tsc && vite build`。

## 5. 重要业务约定

### 案例库

- 案例库是门店真实交付资产。
- 案例可以涉及案例币、审核、公开分享、客户留资。
- 客户案例和送货实拍不应该混进素材库的小筛选里。
- 案例公开前必须关注隐私：人脸、门牌号、手机号、地址、合同、价格、竞品品牌、杂乱现场等。

### 素材库

- 素材库不是案例库的重复入口。
- 产品资料用于导购快速了解“这是什么产品、适合谁、解决什么痛点”。
- 朋友圈 / 小红书内容可以作为素材内容存在，但产品资料卡片下方不需要再放“朋友圈 / 小红书”按钮。
- 不需要单独做“销售话术”主分类。
- 不放客户案例、送货实拍、案例币逻辑。

### 产品资料

当前产品资料筛选方向：

- `全部`
- `怡然`：包含怡然和怡风，不单独显示怡风筛选
- `芸枫`：FORM 和芸枫是同一产品口径，前台不要单独显示 FORM 筛选
- `Living`
- `枕头`
- `智能床`

命名约定：

- 怡然系列前台按软硬度命名：`怡然-软`、`怡然-中`、`怡然-中偏硬`、`怡然-硬`
- 不按 `星柔 / 星空 / 星妙 / 星钻` 作为前台主命名
- `千禧感温枕` 已改名为 `感温舒颈枕`
- 前台不显示来源文件备注
- ONE 尚系列是老款，已不建议展示

产品图片要求：

- 产品图片必须尽量与型号对应。
- 不确定的图片宁可使用统一占位图，也不要乱配相似型号。
- PDF 提取图清晰度可能不够，后续更适合让管理员在后台上传真实产品图。

## 6. 当前数据存储风险

这是当前真实试用最大风险。

项目目前大量数据仍在浏览器 `localStorage`：

- `sct-user`：登录用户
- `sct-share-leads`：客户留资
- `sct-delivery-tasks`：交付 / 案例任务
- `sct-case-coin-records`：案例币流水
- `sct-point-records`：积分流水
- `sct-delivery-points`：交付积分
- `sct-deal-reports`：成交喜报
- `sct-daily-posting-{userId}`：每日发朋友圈状态
- `sct-xhs-posting-{userId}`：每日小红书状态
- `sct-admin-product-override-{id}`：产品资料覆盖
- `sct-admin-products-created`：管理员新增产品

这意味着：

- A 手机上传的案例，B 手机 / 管理员电脑可能看不到。
- 客户留资只保存在访问者或当前浏览器里，真实业务会丢线索。
- 管理员改产品资料，只在自己的浏览器生效。
- 清缓存会丢失新增案例、线索、产品修改和积分记录。

如果目标是真实试用，P0 是接真实后端或 Supabase/Firebase 这类云数据库。不要继续把关键业务建立在 localStorage 上。

## 7. 当前已发现的待优化点

优先级建议：

### P0：真实试用数据同步

- 案例任务、安装图、成交故事、审核状态、客户线索、产品资料管理必须迁到后端。
- 管理员后台必须能跨设备看到导购和客户提交的数据。
- 客户留资必须服务端落库，不可只写本机 localStorage。

### P0：导购核心路径

- 从产品筛选进入详情后，返回应回到原筛选，例如从 `Living` 进入详情，返回仍停留在 `Living`。
- 产品资料卡片下方不需要 `朋友圈 / 小红书` 两个按钮。
- 产品资料筛选状态建议写进 URL，例如 `/library?cat=product&series=Living`。

### P1：首页信息优先级

- 导购首页功能较多，真实使用时建议聚焦三件事：
  - 补成交故事
  - 分享可公开案例给客户
  - 上传 / 创建新案例
- 朋友圈、小红书、成交喜报、积分可以放低层级，不要抢主任务。

### P1：客户分享页与留资

- 分享页方向正确，但要接真实顾问微信 / 电话配置。
- 手机号不能只判断长度，应做基本中国手机号格式校验。
- 留资要记录来源案例、来源导购、来源门店、来源按钮。

### P2：资源与品牌稳定性

- 登录页 TEMPUR logo 目前依赖官网外链，建议改成本地资源。
- 产品图片要继续核对型号对应关系。
- 大图需要压缩，减少手机首屏和详情页加载压力。

## 8. 后端第一版建议

如果开始接真实后端，建议先做最小可用数据层，不要一次做复杂系统。

第一批核心表：

- `users`：用户、角色、门店归属
- `stores`：门店
- `products`：产品资料
- `product_images`：产品多图
- `delivery_cases`：案例 / 交付任务
- `case_images`：案例图片
- `leads`：客户线索
- `point_records`：积分 / 案例币流水

第一批 API：

- 登录 / 获取当前用户
- 产品列表 / 详情 / 新增 / 修改 / 上传图片
- 创建案例任务
- 上传案例图片
- 补充成交故事
- 管理员审核案例
- 公开分享页读取案例
- 客户提交留资
- 管理员查看线索

图片存储：

- 不要把正式图片继续存 dataURL。
- 使用 OSS / COS / Supabase Storage / S3 类对象存储。
- 前端只保存图片 URL 和元数据。

## 9. 前端修改注意事项

- 编辑文件时优先保持现有 React + Tailwind 风格，不引入新 UI 框架。
- 图标继续使用 `lucide-react`。
- 手机 H5 是核心体验，改完必须检查窄屏显示。
- 不要做营销 landing page；首屏应是可用工作台。
- 不要把卡片层层嵌套成过重 UI。
- 文案要适合导购、门店老板、安装师傅的真实工作语言，少写抽象说明。
- 公开客户页不要暴露客户真实姓名、电话、详细地址、门牌、合同、价格等敏感信息。
- 如果修 copy 功能，要考虑微信 / 小红书内置浏览器，保留 `copyText` 的降级弹窗逻辑。

## 10. 验收清单

每次重要修改后至少运行：

```bash
npm.cmd exec tsc -- --noEmit
npm.cmd run build
```

导购账号验收路径：

1. 登录 `sales001 / 123456`。
2. 刷新 `/`、`/library`、`/profile`，确认不丢登录态。
3. 进入 `/library?cat=product`，测试搜索和筛选。
4. 从 `Living` 或 `枕头` 筛选进入产品详情，返回后仍保留筛选。
5. 进入 `/cases-hub`，检查待补故事、可分享案例、复制链接、预览客户页。
6. 从客户分享页提交留资，确认管理员后台可见。
7. 进入 `/delivery/create`，创建案例采集任务，确认安装师傅 / 管理员可见。

管理员账号验收路径：

1. 登录 `admin / admin123`。
2. 进入 `/admin/products`，新增或修改产品资料和多图。
3. 回到导购账号刷新素材库，确认产品资料同步。
4. 进入 `/admin/delivery`，审核案例。
5. 进入 `/admin/leads`，查看客户留资。

## 11. 和用户协作方式

用户经常会让 Codex 先审视、给指令，再让 Claude Code 执行；也可能直接让 Codex 修改。接手时要先判断用户当前是在要“建议/指令”还是“直接实现”。

用户偏好的输出：

- 中文
- 结构清晰
- 能直接复制给 Claude Code 的指令
- 不要空泛，要说清楚为什么改、改哪里、怎么验收

如果用户说“不要改代码，只查问题”，只能阅读和给建议。
如果用户说“PLEASE IMPLEMENT THIS PLAN”，通常可以直接修改并验收。
